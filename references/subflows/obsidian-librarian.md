# 子流程：Obsidian 能力库管家

> 总控在需要把来源、蒸馏记录、能力卡、候选 Skill、模拟测试结果或真实应用反馈
> 安全归档到 Obsidian 能力库时读取本文件并按其规则执行。归档脚本位于
> `scripts/archive_to_obsidian.py`，随本 Skill 自包含。

## 绝对禁止

- 不删除既有笔记；
- 不覆盖内容不同的既有笔记（归档脚本遇此情况抛 `FileExistsError`）；
- 不静默合并内容；
- 写入失败时生成待归档清单，支持幂等重试。

每次归档写入 YAML 元数据（唯一编号、标题、主/辅助领域、深度、阶段状态、成熟度、来源链接、
内容属性、Skill 版本、验证状态、关系、标签），并先检索已有节点再决定创建或建立关系。

## Vault 目录结构

Obsidian 能力库根目录：`E:\Obsidian\Skill_Library`

必需目录：

| 目录 | 用途 |
|---|---|
| `00_能力地图` | 个人能力树、能力域索引、最近更新 |
| `01_来源资料` | 原始资料副本或来源指针 |
| `02_蒸馏记录` | 阶段状态、蒸馏过程记录 |
| `03_能力卡片` | 判断卡、SOP、检查清单、模板 |
| `04_Skill候选` | 未通过模拟测试的候选 Skill |
| `05_Skill已验证` | 通过模拟测试并经确认的 Skill |
| `06_测试记录` | 模拟测试用例与结果 |
| `07_应用反馈` | 真实应用记录、偏差与修订 |
| `90_模板与配置` | 模板与备份 |

每项资产至少保存共同元数据：唯一编号、标题、创建/更新时间、资料类型、主领域、辅助领域、
处理深度、当前阶段状态、来源链接、内容属性（原话/原意/AI 重构/个人应用）、成熟度、
Skill 版本与验证状态、关系、标签、待核实事项。

发现同名或近似内容时不覆盖原笔记，保留两个来源并建立重复/补充/增强/冲突关系，
只有用户明确确认后才允许人工合并。

### 能力域子目录（推荐，2026-07-06 起新增）

为避免不同主领域（audit / ai / cognition / general / content / side-business）的资产在同一目录混杂导致检索与加载干扰，**推荐按主领域建立子目录**：

```
03_能力卡片/
├── audit/         ← 审计域能力卡（如 CA-001~016、CB-001~030）
├── ai/            ← AI 域能力卡（未来）
├── cognition/     ← 认知域
├── content/       ← 内容域
├── general/       ← 通用域
└── side-business/ ← 副业域

05_Skill已验证/ 和 04_Skill候选/、06_测试记录/、07_应用反馈/ 同理按主领域分子目录。
```

**子目录判断规则**（由 obsidian-librarian 执行）：

1. **已建主领域子目录时**：根据新资产的 `primary_domain` 字段落到对应子目录。
   - 主领域 `audit` → `03_能力卡片/audit/`
   - 主领域 `ai` → `03_能力卡片/ai/`
   - 主领域 `general` → `03_能力卡片/general/`
   - 其他类推。
2. **仍为顶层扁平结构时**：新资产落到顶层目录（如 `03_能力卡片/`），向后兼容。
3. **跨主领域资产**（如"AI 提效审计"）：仅放在主领域对应的子目录，`auxiliary_domains` 字段做引用关系而非重复创建。
4. **主领域难以判断时**：追问用户而非默认归类。

**子目录导航页约定**：每个已建子目录下放 `README.md` 作为资产清单 + 使用建议入口，便于人类和 AI 快速定位。

**子目录迁移策略**：当能力库从扁平升级到子目录结构时，应一次性迁移全部现有资产并更新所有引用路径（详见 `references/subflows/obsidian-librarian.md` 中的迁移步骤）。

## 能力树节点模型

能力树是运行时检索结构，不是资料目录。

- 根：长期问题、核心目标、个人优势；
- 枝干：通用、审计、AI、副业、内容、认知；
- 叶子：来源资料、案例、复盘、能力卡、Skill。

关系类型：重复、补充、增强、冲突、上游、下游。

规则：

- 生成新资产前必须先检索已有节点，避免重复生产并优先复用已验证能力；
- 每个节点记录来源、解决的问题、调用场景、关系、成熟度、对应 Skill 和实际使用记录；
- 应用反馈必须与所用 Skill 版本关联，修订产生新版本并重新测试。

写入失败时生成待归档清单（目标路径、失败原因、重试状态），下次运行先检测并提示安全重试；
重试保持幂等，不重复生成同一资产。
